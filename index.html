AI 邮件自动生成与智能跟进系统

外包客服每天花 3-5 小时手动编写英文邮件回复异常订单 → AI 全自动替代，从读取订单、查客户邮箱、生成邮件到 24 小时后自动跟进，邮件环节无需人工参与。

2026.04 - 2026.05 · 已上线稳定运行
▼
📌 业务背景

跨境电商平台每日产生大量物流异常订单（海关查验、运输延误、地址异常等）。原流程需外包客服逐条登录 Intercom 手动编写英文邮件发送给客户，再回填 Case ID 至飞书表格。每单处理约 15-20 分钟，日处理量 15 单左右，存在漏跟、晚跟、回复质量不一致等问题，人力成本高。

💡 我的方案
  • 业务规则数字化：将人工客服决策链路拆解为结构化决策树——Solution 状态分流、Intercom 去重、敏感词审核、退款拦截——通过 Prompt 引导 AI 生成精确的分流代码
  • 两级 Prompt 设计：第一级从 Solution 中强制提取客户可见事实，过滤内部 SOP 指令；第二级套用历史客服模板润色，确保语气专业、信息准确。支持英/法/西/德四语自适应
  • 跨系统集成：飞书 Base 数据读写 → Shopify Admin GraphQL 订单查询 → Intercom REST API 邮件发送，打通三个系统数据流转
  • 24h 自动跟进：发送后自动检测客户是否回复，未回复则发送二次跟进或转人工
🛠️ 落地实施
  • 先测试环境验证 59 条订单分流准确性，再迁移至正式表批量运行
  • 经多轮迭代修复：Intercom 重复检测（从精确匹配改为包含匹配）、状态刷新、Pending 兜底等问题
  • 配套文档与培训：编写操作手册与字段说明文档，培训外包客服通过飞书审核视图确认 AI 草稿并一键发送
  • 建立 dry-run → 测试环境 → 测试发送 → 正式上线四层安全门控，上线零事故

自动化巡检与飞书预警智能 Agent

自动化系统越来越多、各自独立运行、没人盯着。搭了一套"监控自动化的自动化"——每 5 分钟巡检一次，出故障自动分类、能自愈的自动补跑、不能自愈的即时飞书告警。

2026.05 - 至今 · 已上线稳定运行
▼
📌 业务背景

随着业务自动化项目持续增多，各类自动化任务独立运行、无统一监控入口，任务故障、停运问题无法及时感知——自动化"跑起来"但"没人盯着"，长期存在运维盲区。故障从发生到被发现往往需要数天时间。

💡 我的方案
  • 常驻巡检 Agent：基于 macOS 定时机制搭建，每 5 分钟扫描全量自动化日志，智能校验执行状态（成功/失败/未运行）
  • 智能故障分类：Agent 自主识别网络超时、权限过期、第三方接口异常、脚本语法错误等故障类型，匹配对应标准化处理方案
  • 自愈机制：对可自愈的失败类型（网络闪断、定时未触发等）自动触发补跑并回报结果；不可自愈的即时飞书告警
  • 飞书联动：每日 18:30 自动汇总运行日报推送飞书。支持自然语言交互——在飞书问"今天自动化怎么样"立即出汇总，说"补跑一下缺货"即可触发补跑
  • Web 控制台：搭建本地仪表盘，集中展示所有自动化任务的运行详情、配置、日志路径

Shopify 订单风控判定引擎

运营人员每单花 10-15 分钟手工判断是否欺诈 → AI 全自动 15-30 秒输出结论，7 路并行数据采集、10 维评分、LLM 研判、三级决策融合，每单输出可追溯的风控报告。

2026.03 - 2026.06 · 已上线稳定运行
▼
📌 业务背景

跨境电商 DTC 场景中，信用卡盗刷、转运仓批量下单、身份伪造等欺诈行为频发。原风控流程依赖运营人员逐单登录 Shopify 后台查看风险等级、手动核查地址和 IP 信息，每单耗时 10-15 分钟，人工判断标准不统一，高风险订单漏判时有发生。

💡 我的方案

对标 Dify/LangChain Agent 工作流架构理念，设计了一套多 Agent 协作的完整 Pipeline：

  • 7 路并行数据采集：Shopify 订单 / IP 地理定位 / 邮箱信誉与社交档案 / 地址地理编码与深查 / 身份交叉验证 / 电话验证——任一采集失败不影响整体判定，15-30 秒输出结论
  • 硬规则引擎：业务风控策略代码化为确定性规则——历史欺诈关联订单直接拦截、Shopify high risk 自动拦截、$0 订单+多件商品拦截。规则优先级硬编码，不受 LLM 影响
  • 10 维评分模型：价格异常、历史欺诈、身份弱信号、地址异常、IP 风险、商品结构、支付验证、行为特征、排重关联、综合信号——输出 0-100 风险分
  • LLM 深层研判：识别"一次性邮箱+转运仓地址+首单+高价值"等多信号组合，区分 VPN 出差等误报与真实欺诈链
  • Agent 主动补查：中高风险订单自动触发 Google 街景验证和身份搜索，结合 Playwright 浏览器自动化采集实景证据
  • 三级决策融合：硬规则 > LLM 研判 > 评分模型，PASS/REVIEW/REVIEW_BLOCK/BLOCK 四档决策

10-15 min → 15-30s单笔判定时间
~95%效率提升

G+ 平台自动拆单系统

异常订单拆单需要人工判断 + 手动操作浏览器 → 全自动拆单：自动鉴权、从 MySQL 拉异常单、匹配库存出方案、Playwright 自动执行拆单、结果回填飞书。

2026.06 · 已上线
▼
📌 业务背景

G+ 平台异常订单需要人工判断是否拆单，再手动在浏览器中操作拆单——步骤繁琐、耗时长、容易出错。

💡 我的方案

三阶段全自动链路：

  • bootstrap:auth：首次或鉴权失效时自动登录 G+，缓存接口鉴权凭证
  • plan:inventory：从 MySQL 拉取异常订单 → 库存匹配运算 → 生成拆单方案 → 写入飞书表格
  • run:sheet-queue：对状态为"待拆"的订单，用 Playwright 浏览器自动化执行拆单操作

拆单成功/失败/原因自动写回飞书，可追溯可审计。

缺货订单双源排查系统

客服每天从 G+ 和 WMS 两个系统反复查缺货、对库存、做 Excel → 自动化每天跑 3 次，双源对比 + 二次复核降误报，结果同步飞书并邮件通知。

2026.04 - 至今 · 每天 09:00 / 14:00 / 17:00 自动运行
▼
📌 业务背景

G+ 系统里有很多异常订单，但"异常"不等于"缺货"——有些只是系统延迟，库存其实够。WMS 仓储也有待发货的异常缺货单，两个系统没打通。客服每天要打开两个系统查单、对库存、做 Excel、发邮件、回填飞书，效率低且容易漏。

💡 我的方案
  • G+ 缺货识别：从 G+ 抓异常订单和 SKU 明细 → 查真实库存，按收货国家区分缺货逻辑（美国：洪梅仓=0 且美国自建仓=0 才算缺货；非美国：洪梅仓=0 即缺货）
  • WMS 补充排查：从 WMS 筛出待发货/分拣中订单 → 找出预锁定库存为 0 的 SKU → 补充 G+ 订单号等信息
  • 二次复核："异常单"不等于"缺货单"，双重逻辑过滤减少误报
  • 飞书同步：自动写入飞书多维表格，高亮优先级，维护待处理/已处理状态
  • 邮件摘要：每天 3 次定时运行后自动发送摘要通知
📊 量化成果
3 次/天自动运行
~90%人工节省
双源G+ + WMS
2 层误报过滤
