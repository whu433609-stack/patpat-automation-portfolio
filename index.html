缺货订单双源排查系统

客服每天从 G+ 和 WMS 两个系统反复查缺货、对库存、做 Excel → 自动化每天跑 3 次，双源对比+二次复核，自动找出真正缺货的订单。

基础层 · 2026.04 - 至今 · 每天 09:00/14:00/17:00
▼
背景 & 痛点

G+ 系统中有大量异常订单，但"异常"不等于"缺货"——有些只是系统延迟。WMS 仓储也有待发货的缺货单。两个系统未打通。客服每天需打开两个系统查单、对库存、做 Excel、发邮件、回填飞书，效率低且易遗漏。

方案架构
实现细节
  • 从 G+ 拉异常订单，查真实库存——美国和非美国订单缺货逻辑不同，分开判断
  • 从 WMS 拉待发货订单中可能缺货的，补充 G+ 订单号等信息
  • 双源数据合并后二次复核，过滤误报
  • 结果写入飞书多维表格，自动标记优先级和超时订单
  • 每天 3 次定时运行，执行后邮件摘要通知相关人员
搭建难点与自研解法
🔸 难点 1：G+ 和 WMS 的订单编号格式不统一，关联匹配困难

建立统一订单映射表，通过 SKU+数量+时间窗口做模糊匹配，再用飞书 Base 的关联字段做人工确认入口。

🔸 难点 2："异常"不等于"缺货"，大量系统延迟误报

设计两层过滤——先按库存阈值粗筛（库存低于 120% 订单量为候选），再对候选单查真实可用库存做二次复核。

🔸 难点 3：美国仓和非美国仓的缺货判断逻辑不同

通过订单的 ship_country 字段分流，维护两套库存获取接口，各走各自的判断逻辑。

方案架构
实现细节

🔸 难点 2：Intercom 发送后缺少回调机制，无法自动追踪客户回复

🔸 难点 3：测试环境和正式环境的数据隔离

通过环境变量控制 base URL 和数据表名，写操作和发信操作各自独立开关控制。上线前首先经过 dry-run 验证，确保正式数据不会被测试流程污染。

方案架构
实现细节

  • 等待 30 分钟缓冲，避免 G+ 和 Shopify 的状态同步延迟导致误报
  • 通过 Shopify Admin API 查询这些订单的实际发货状态
  • 交叉比对逻辑：G+ 标发失败但 Shopify 显示已发货 → 标记为"系统延迟"不预警；真正失败的按优先级发邮件

🔸 难点 1：G+ 和 Shopify 的订单状态有时间差，导致大量误报

从 G+ 获取订单后等待 30 分钟再查 Shopify 状态，利用时间窗缓冲消除系统延迟带来的误报。上线后误报过滤率约 70%。

🔸 难点 2：双店铺的数据隔离和配置管理

🔸 难点 3：每次查询需要调用 Shopify API，频率过高可能触发限流

每天只跑两次（09:00/17:00），每次批量查询。标发失败不是高频事件，两次/天的频率完全满足运营的响应时效要求（最晚 2h 内发现）。

每周自动从 Shopify 拉取退款数据，按周对比退款率变化，生成 Excel 报表并通过飞书推送给管理层。

▼
背景 & 痛点

运营需要每周手动拉取 Shopify 退款数据，做成 Excel 报表汇报给管理层。手工操作耗时约 2 小时/周，且数据口径不统一——有的人看退款率，有的人看退款金额占比。

方案架构
实现细节
  • 通过 Shopify Admin API 拉取指定时间段的订单数据
  • 先检查 API 权限，权限不足时自动降级为基础报表（仅返回摘要数据，不影响主流程）
  • 按周对比退款率变化（本周 vs 上周 vs 上月同期）
  • 报表内容：退款率趋势、退款原因分类、退款原因 Top10、取消原因 Top10
  • Excel 报表自动通过飞书机器人推送给对接人
搭建难点与自研解法
🔸 难点 1：Shopify API 有速率限制（leak bucket 算法），大量订单时可能超限

实现请求队列 + 指数退避重试策略。配额不足时自动降级到基础报表，只返回关键指标而非全量明细。

🔸 难点 2：退款口径不统一——按金额算 vs 按单数算

报表同时输出金额维度退款率和单数维度退款率，并标注管理层关注哪个口径。Excel 公式预设好，打开即可看到最终结论。

🔸 难点 3：历史数据量大，首次全量拉取耗时过长

首次全量后每日增量更新，周报只拉取最近 4 周数据做对比。月度/季度汇总通过离线聚合表预计算。

AI 邮件自动生成 & 智能跟进系统

外包客服每天花 3 小时手动写英文邮件回复异常订单 → AI 全自动替代，从读订单、查邮箱、生成邮件到 24 小时后自动跟进，邮件环节人工彻底解放。

进阶层 · 2026.04 - 2026.05 · 已上线稳定运行
▼
背景 & 痛点

跨境电商平台每天有大量物流异常订单——海关查验、运输延误、地址异常等。外包客服需要一条条登录 Intercom 手动写英文邮件发给客户，再把 Case ID 回填到飞书表格。每单 15-20 分钟，日处理 15 单左右，经常漏跟、晚跟，回复质量也参差不齐。

方案架构
实现细节
  • 将客服决策链路（状态分流、Intercom 去重、敏感词审核、退款拦截）拆解为规则树，通过 Prompt 引导 AI 生成分流代码
  • 两级生成策略：第一级从工单记录中提取客户可见的事实信息（如"包裹编号""异常原因"等），输出结构化 JSON；第二级套用历史客服模板润色，确保语气专业。支持英文、法文、西班牙文、德文
  • 打通三个系统：飞书 Base 读取订单 → Shopify 查询客户邮箱 → Intercom 发送邮件
  • 建立四层安全机制：dry-run → 测试环境 → 测试发送 → 正式上线，确保零事故
  • 配套操作手册，培训外包同事通过飞书审核视图确认草稿
搭建难点与自研解法
🔸 难点 1：多语言邮件生成质量不一致，法文和德文经常出现语法错误

采用"两级生成"——第一级抽取事实（中英文混杂的工单数据 → 结构化字段），第二级用目标语言套模板润色。模板固化常见句式，AI 只填充变量部分，语法问题减少 80%+。

🔸 难点 2：Intercom 发送后如何确认邮件真正送达和客户回复？

Intercom 的 API 会返回 message ID。在飞书记录 message ID，跟进检测时通过 Intercom 查询该 conversation 是否有客户回复。不依赖邮件送达率（Intercom 内部会重试），只看是否有回复。

🔸 难点 3：四层安全机制每层如何确保不遗漏正式上线？

第一层 dry-run（只打印日志，不写表不发信）→ 第二层测试环境（写测试表，发测试 Intercom）→ 第三层测试发送（写正式表，只发邮件到测试邮箱）→ 第四层正式上线。每层有对应的 flyway 迁移脚本回滚。上线初期该机制至少拦截了 3 次潜在问题。

G+ 平台自动拆单

异常订单拆单需要人工判断+手动操作 → 全自动三阶段：鉴权、出方案（库存匹配）、Playwright 自动执行，结果回填飞书。

进阶层 · 2026.06 · 已上线
▼
背景 & 痛点

G+ 平台的异常订单需要人工判断是否拆单，确定方案后再手动在浏览器中操作拆单。步骤多、耗时长、容易出错。

方案架构
实现细节
  • 自动鉴权：首次或鉴权失效时自动登录 G+，缓存接口凭证，本地加密存储
  • 出方案：从 MySQL 拉取异常订单 → 库存匹配运算（约束参数化：最大拆分数量、最小单品数、发货时效阈值）→ 生成拆单方案 → 写入飞书表格
  • 自动执行：对状态为"待拆"的订单，用 Playwright 浏览器自动化执行拆单操作，支持断点续执行
  • 结果自动写回飞书：拆了哪些 SKU、未拆原因，可追溯可审计
搭建难点与自研解法
🔸 难点 1：G+ 平台的鉴权状态会不定期过期，导致拆单执行失败

设计鉴权前置阶段——每次任务开始先检测缓存凭证是否有效，失效则自动触发 Playwright 重登录。登录凭据本地加密存储，支持人工手动刷新入口。

🔸 难点 2：库存匹配方案需要考虑多种约束，最优解不易计算

将约束条件参数化（最大拆分数量、最小拆单单品数、发货时效阈值等），用 MySQL 查询 + 业务规则引擎计算出最优方案。如果最优方案不满足约束，降级为方案解释供人工参考。

🔸 难点 3：Playwright 执行过程中页面元素变化（G+ 平台更新）

为关键操作步骤添加 fallback selector。当主选择器匹配失败时，按优先级尝试备用选择器，记录所有失败的 DOM 快照便于定位升级后的变更点。

Meta 评论 AI 自动回复

通过 Meta Graph API 拉取 Facebook/Instagram 评论，AI 五类分类+生成回复建议，飞书审核后自动/手动回复。

进阶层 · 2026.03 - 至今 · 每 30 分钟自动运行
▼
背景 & 痛点

Facebook 和 Instagram 的客户评论需要人工逐条查看、分类、撰写回复。每天 50-100 条评论，运营需花 1-2 小时处理。投诉类评论回复不及时可能影响品牌形象。

方案架构
实现细节
  • 通过 Meta Graph API 定时拉取 Facebook Page 和 Instagram 的客户评论
  • AI 自动分为 5 类：咨询/投诉/好评/退换货/其他
  • 分类后 AI 自动生成回复建议（英文），包含主要内容和语气建议
  • 结果写入飞书多维表格，运营在视图中审核并决定"用 AI 回复"或"自己重写"
  • 好评类配置自动回复白名单直接回复，其余需审核，降低风险
搭建难点与自研解法
🔸 难点 1：Meta Graph API 的访问令牌会过期（60 天有效）

设置令牌过期检测，提前 7 天通过飞书提醒人工续期。已实现令牌过期时的降级处理——不中断已有分类，只暂停新数据拉取。

🔸 难点 2：投诉类评论的回复建议需要格外谨慎，AI 可能措辞不当

投诉类默认不走自动回复，全部进入飞书审核。AI 仅生成"建议版本"供参考，运营可以一键采用或修改。上线后投诉建议采用率约 50%，好评类采用率超 90%。

🔸 难点 3：拉取频率和内容去重——同一评论不要反复处理

在飞书 Base 中记录已处理的评论 ID，每次拉取后先做 ID 去重。新增评论才进入分类流程。拉取间隔设为 30 分钟，兼顾时效性和 API 调用成本。

单条从 5-10 分钟→AI 30 秒生成 · 日均 50-100 条 · 投诉响应从 2h→10 分钟 · 好评自动 30%

跨境电商 DTC 场景中，信用卡盗刷、转运仓批量下单、身份伪造等欺诈行为频发。风控人员需逐单登录 Shopify 查看风险等级，手动核查地址、IP、邮箱等，每单 10-15 分钟。人工判断标准不统一，高风险订单时有漏判。

方案架构
实现细节
  • 7 路并行采集：订单信息、IP 归属地、邮箱信誉、地址真实性、身份信息、电话验证、历史记录——任一采集失败不影响整体判定
  • 硬规则兜底：历史欺诈关联订单直接拦截、Shopify high risk 自动拦截、$0 订单+多件商品拦截——不受 AI 影响，保证底线安全
  • 多维度评分+AI 研判：价格异常、地址异常、IP 风险、支付验证等 10 个维度综合评分，再由 LLM 做最终判断
  • 中高风险订单自动触发补查（地址验证 API、地理编码服务等），实景证据存入飞书附件
  • 每单输出结构化报告：命中规则、风险评分、AI 分析、关键证据，运营 5 秒可理解判定依据
搭建难点与自研解法
🔸 难点 1：7 路数据源并行采集时，部分数据源超时或失败影响整体判定速度

采用 asyncio.gather() + 超时控制。单个数据源超过 5 秒就算作"数据不可得"，判定时标记该维度为"缺失"，不影响其他维度的评分和最终的 LLM 判定。

🔸 难点 2：LLM 的判定结果可能不一致（同一订单多次判断结果不同）

对 LLM 输出做结构化约束（限定为 PASS/REVIEW/BLOCK+理由），再套一层规则校验：如果 LLM 判定结果与硬规则引擎冲突，取更严格的结论——风控场景下"误放"成本远高于"误拦"。

🔸 难点 3：某些场景需要补查实景证据（如 Google 街景验证地址），手动做太慢

通过 LLM 自动判断是否触发补查。当"地址真实性"维度得分低于阈值时，自动调用地址验证 API + 地理编码服务，采集的实景截图存入飞书附件供运营参考。

方案架构
实现细节
  • 基于 macOS 定时机制搭建常驻巡检 Agent，每 5 分钟扫描全量日志，判断任务成功/失败/未运行

搭建难点与自研解法
🔸 难点 1：17 个任务的故障模式各不相同，统一分类困难

建立故障分类树——网络超时/权限过期/接口异常/脚本错误/数据异常等 5 大类，每类关联对应的处理方案。分类器先用规则匹配（错误码+日志关键词），匹配不到的 fallback 到 LLM 分析。

🔸 难点 2：自愈操作需要考虑幂等性，不能重复执行导致数据异常

每个任务记录"上次自愈时间"和"自愈次数"。同一任务 30 分钟内不重复自愈，连续 3 次自愈失败自动升级为人工告警。

🔸 难点 3：自然语言查询的意图识别准确率起步只有 60%

先用预设意图模板覆盖 80% 高频查询（"今天状态""补跑缺货""最近故障"等），剩余 20% 走 LLM 意图识别。模板匹配 200ms 内响应，LLM 路径约 3 秒。

方案架构
实现细节
  • 接入 DeepSeek 模型，支持多轮对话
  • 核心指令：订单查询、系统状态查询、任务补跑、知识库问答

搭建难点与自研解法
🔸 难点 1：多 Agent 调度——如何判断用户意图该用哪个工具

设计"意图-工具"映射表。LLM 先判断意图类别，再路由到对应工具执行。如果意图不明确，用追问引导用户（"您想查订单还是看系统状态？"）。

🔸 难点 2：对话成本控制——每次都用 LLM 处理太贵

6 个核心高频指令走模板匹配（覆盖率 95%），仅在匹配不到时 fallback 到 LLM。DeepSeek 按量计费，日调用 200-300 次，月度成本控制在几十元。

🔸 难点 3：安全性——机器人不能成为系统后门

工具调用做两层限制：① 只有白名单群聊中的消息才会被处理；② 高危操作需要二次确认。巡检 Agent 有兜底，就算误操作也能 5 分钟内发现异常。
