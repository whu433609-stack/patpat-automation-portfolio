AI 邮件自动生成与智能跟进系统

已上线 2026.04 - 2026.05 Python · Intercom · Shopify · 飞书 Base
🎯 背景 & 痛点

跨境电商平台每日产生大量物流异常订单（海关查验、运输延误、地址异常等），原流程需外包客服逐条登录 Intercom 手动编写英文邮件发送给客户，再回填 Case ID。每单约 15-20 分钟，日处理量约 15 单，存在漏跟、晚跟、回复质量不一致等问题，人力成本高。

🏗️ 系统架构
⚙️ 我是怎么实现的
  • 业务规则数字化：将人工客服决策链路（Solution 状态分流、Intercom 去重、敏感词审核、退款拦截）拆解为结构化决策树，通过 Prompt 引导 AI 生成精确的分流代码
  • 两级 Prompt 设计：第一级从 Solution 中强制提取客户可见事实，过滤内部 SOP 指令；第二级套用历史客服模板润色，确保信息准确且语气专业。支持 英/法/西/德 四语自适应
  • 跨系统集成：飞书 Base 数据读写、Shopify Admin GraphQL 订单查询、Intercom REST API 邮件发送——打通三个系统数据流转
  • 灰度上线四层门控：dry-run → 测试环境 → 测试发送 → 正式上线，确保所有自动化项目上线零事故
📊 成果
15→0 min人工处理/单
~30%释放客服人力
10+ 类物流场景覆盖
4 语自动适配
0 事故灰度上线

自动化巡检与飞书预警智能 Agent

已上线 2026.05 - 至今 Python · 飞书 · launchd · Web 控制台
🎯 背景 & 痛点

随着业务自动化项目持续增多，各类自动化任务独立运行、无统一监控入口，任务故障、停运问题无法及时感知——自动化"跑起来"但"没人盯着"，长期存在运维盲区。

🏗️ 系统架构
⚙️ 我是怎么实现的
  • 智能巡检核心：基于 macOS launchd 搭建常驻巡检 Agent，每 5 分钟扫描全量自动化日志，智能校验执行状态（成功/失败/未运行）
  • 故障智能分类：Agent 自主识别网络超时、权限过期、第三方接口异常、脚本语法错误等故障类型，匹配对应标准化处理方案
  • 自愈机制：对可自愈的失败类型自动触发补跑并回报飞书；不可自愈的即时告警
  • 飞书联动：自动汇总数据写入飞书"自动化巡检"表格，18:30 推送日报。飞书直接问"今天自动化情况"即时汇总，说"补跑一下缺货"即可触发补跑
  • Web 控制台：搭建本地仪表盘，展示所有自动化任务运行详情和资产台账（plist 配置、PID、退出状态、日志路径）
📊 成果
5 min巡检间隔
数天→实时故障发现周期
17纳管任务
6400+ 行运行日志
📝 工作记录

系统日志已积累 6400+ 行运行记录，注册表中 17 个自动化任务处于统一管控中，Web 控制台页面可查看每个任务的 plist 配置、PID、退出状态、日志路径等信息。控制台会把 launchd 规则、入口命令、工作目录、日志路径、当前 PID/最近退出状态集中展示。

Shopify 订单风控判定引擎

已上线 2026.03 - 2026.06 Python · LLM · Shopify · Playwright
🎯 背景 & 痛点

跨境电商 DTC 场景中，信用卡盗刷、转运仓批量下单、身份伪造等欺诈行为频发。原风控流程依赖运营人员逐单登录 Shopify 后台查看风险等级、手动核查地址和 IP 信息，每单 10-15 分钟，判断标准不统一，高风险订单漏判时有发生。

🏗️ 系统架构
⚙️ 我是怎么实现的
  • 7 路并行数据采集：Shopify 订单 / IP 地理定位 / 邮箱信誉与社交档案 / 地址地理编码与深查 / 身份交叉验证 / 电话验证——任一采集失败不影响整体判定，15-30 秒输出结论
  • 硬规则引擎：业务风控策略代码化为确定性规则——历史欺诈关联订单直接拦截、Shopify high risk 自动拦截、$0 订单+多件商品拦截、中等风险标记复核
  • 10 维评分模型：价格异常、历史欺诈、身份弱信号、地址异常、IP 风险、商品结构、支付验证、行为特征、排重关联、综合信号 → 输出 0-100 风险分
  • LLM 深层研判：识别"一次性邮箱+转运仓地址+首单+高价值"等多信号组合，区分 VPN 出差等误报与真实欺诈
  • 三级决策融合：硬规则 > LLM 研判 > 评分模型，PASS/REVIEW/REVIEW_BLOCK/BLOCK 四档决策，每单输出可追溯报告
📊 成果
~95%效率提升
15-30s单笔判定
10判定维度
7路并行数据源

G+ 拆单自动化

已上线 2026.06 Node.js · Playwright · MySQL · 飞书
🎯 背景 & 痛点

G+ 平台异常订单需要人工判断是否拆单，再手动在浏览器中操作拆单——步骤繁琐、耗时长、容易出错。

⚙️ 实现逻辑
⚙️ 实现细节
  • 接口层鉴权：自动登录 G+ 并缓存接口鉴权凭证，无需人工维护
  • 拆单计划引擎：自动从 MySQL 拉取异常订单 → 库存匹配运算 → 生成拆单方案 → 写入飞书表格
  • Playwright 自动化执行：自动打开浏览器，对状态为"待拆"的订单执行拆单操作
  • 结果回填：拆单成功/失败/原因自动写回飞书，可追溯可审计
